# 기능 구현 잠재 이슈 및 초보자 문구 점검 (2026-05-04)

## 점검 범위

참조 문서:

이번 문서는 Android PWA에서 무시드 추천 번호가 고정되던 문제와 기본 PWA 캐시 문제를 수정한 뒤, 그 외 기능 구현상 남아 있을 수 있는 위험과 초보 사용자용 문구 개선 방향을 정리한 후속 점검 문서다. 아래의 "현재" 표현은 감사 당시 화면/문서에서 발견한 개선 전 용어를 포함한다.

## 빠른 결론

- 번호 추천 고정 문제 자체는 런타임 난수 시드와 워커 캐시 버전 갱신으로 재발 방지 장치가 들어갔다.
- PWA 쪽은 서비스워커의 versioned worker/module asset 오프라인 접근, manifest installability, 업데이트 토스트 문구, 캐시 쓰기 실패 처리까지 기본 안정성은 좋아졌다.
- 남은 큰 축은 기능 버그라기보다 "운영/사용성 리스크"다. 데이터 최신성, 동기화 실패 안내, 백업/복원 안전성, 초보자에게 어려운 용어, 모바일 성능 체감, 동적 결과 접근성은 계속 보강할 가치가 높다.
- `README.md`와 `claude.md`는 구현 흐름을 잘 담고 있지만, 초보 사용자용 안내라기보다는 개발 인수인계/변경 기록 성격이 강하다. 사용자-facing 도움말은 별도 짧은 문서나 앱 내부 문구로 분리하는 편이 좋다.

## 2026-05-04 구현 반영 상태

- 사용자-facing 명칭을 `번호 추천`, `내 번호 보관함`, `데이터 연결 주소(고급)`, `같은 번호 다시 만들기 코드`, `분석 강도` 중심으로 교체했다.
- 생성/번호 추천/시뮬레이션에 `빠름/기본/정밀` 분석 강도 프리셋을 추가하고 세부 수치 입력은 `고급 설정`으로 접었다.
- 설정과 최신 당첨 카드에 `내 데이터: N회 / 예상 최신: M회 / 차이 K회` 문구를 표시한다.
- `scripts/check_static_data_freshness.mjs`와 `npm run check:data-freshness`를 추가했고, build 검증 순서는 `lint -> check:data-freshness -> smoke`로 변경했다.
- PWA 업데이트 API(`window.lottoPwaUpdate`)와 설정 모달의 `앱 업데이트 확인/준비된 업데이트 적용` 흐름을 추가했다.
- 백업 가져오기 미리보기, `바꾸기` 전 자동 백업(`lotto_before_replace_*.json`), `백업하고 정리하기` 저장 공간 흐름을 추가했다.
- 주요 결과 영역에 live/busy 접근성 속성을 추가하고, `innerHTML` 사용량 per-file allowlist smoke를 추가했다.
- Android-like PWA 무시드 생성/번호 추천 반복 실행 브라우저 테스트(`npm run test:pwa-mobile`)를 추가했다.
- `README.md`, `claude.md`, `gemini.md`, `deploy_github_pages.md`, `proxy/README.md`의 용어와 검증 명령을 현재 구현 기준으로 맞췄고, `.gitignore`에 로컬 백업 JSON 산출물 제외 규칙을 추가했다.
- IndexedDB 전환은 이번 구현 범위에서 제외했고, 후속 구조 개선 항목으로 유지한다.

## 확인된 상태

- 정적 당첨 데이터 `data/winning_stats.json`은 1220개 행이고, 최신 정적 회차는 `1221회`, 날짜는 `2026-04-25`다.
- 2026-05-04 KST 기준 앱의 `estimateLatestDrawKST()` 로직상 예상 최신 회차는 `1222회`다. 즉 정적 JSON만 보면 1회차 차이가 나며, 앱은 로컬 업데이트/동기화로 이를 보완하는 구조다.
- 서비스워커 캐시 버전은 `v20`로 문서와 코드가 맞춰져 있다.
- `assets/modules/utils/strings.js`는 UTF-8로 정상 표시된다. 단, Windows PowerShell 기본 인코딩으로 읽으면 한글이 깨져 보일 수 있으므로 문서/편집기 기준을 UTF-8로 명시하면 좋다.
- `innerHTML` 렌더링 지점은 앱 코드 곳곳에 남아 있다. 일부는 escape 처리가 되어 있고, 일부는 내부 계산값만 사용하지만, 외부/백업/동기화 데이터가 섞이는 화면은 회귀 테스트를 계속 붙여야 한다.

## 우선순위 높은 개선 후보

### 1. 데이터 최신성 운영 흐름 보강

현재 구조:

- 앱은 정적 JSON을 먼저 로드한다.
- 최신 회차가 부족하면 공식 API/프록시/로컬 업데이트로 보완한다.
- 정적 JSON 실패 시 기존 메모리 데이터를 보존하고, 부분 복구 상태를 표시한다.

잠재 문제:

- 정적 데이터가 1회차 이상 밀릴 때 초보자는 "내 앱이 최신인가?"를 바로 판단하기 어렵다.
- 자동 동기화가 공용 CORS 프록시나 공식 API 상태에 영향을 받기 때문에, 실패 원인이 앱 문제처럼 보일 수 있다.
- `정적 JSON + 로컬 업데이트`, `부분 복구`, `fallback`, `프록시` 같은 표현은 개발자에게는 정확하지만 일반 사용자에게는 어렵다.

권장 추가:

- 앱 상단 또는 설정에 "내 데이터 기준: 1221회까지 저장됨 / 최신 예상: 1222회"처럼 회차 차이를 쉬운 문장으로 표시한다.
- 정적 데이터가 예상 최신보다 뒤처지면 "지금 최신 회차 확인" 버튼을 더 눈에 띄게 노출한다.
- 릴리스 전 `data/winning_stats.json`의 최신 회차와 `estimateLatestDrawKST()`를 비교하는 스크립트를 추가한다. 차이가 1회차를 넘으면 build 또는 배포 체크에서 실패시키는 방식이 좋다.
- 동기화 실패 시 "잠시 후 다시 시도", "오프라인에서도 저장된 번호 확인은 가능", "고급 연결 주소는 비워둬도 됨" 같은 사용자 행동 중심 안내를 함께 보여준다.

### 2. 백업/복원 흐름의 초보자 안전장치

현재 구조:

- 백업 v3 import/export가 있고, Merge/Overwrite 옵션과 테마/프록시/전략/알림 적용 옵션이 있다.
- 파일 크기, 티켓 총량, strategy snapshot 크기 제한이 들어가 있다.
- orphan campaign 정리, 미래 회차 local update 제외 등 데이터 정규화도 있다.

잠재 문제:

- `Merge`, `Overwrite`, `프록시 설정 적용`, `전략 설정 적용`은 초보자에게 손실 범위를 직관적으로 말해주지 않는다.
- 가져오기 버튼을 누르기 전 "무엇이 몇 개 바뀌는지"를 미리 보지 못한다.
- Overwrite는 강력한 동작인데, 실수했을 때 되돌리기 흐름이 명확하지 않다.

권장 추가:

- 가져오기 전 미리보기 모달을 추가한다.
- 예: "즐겨찾기 12개 추가, 티켓 40장 병합, 기존 설정은 그대로 유지"처럼 실제 반영 결과를 먼저 보여준다.
- Overwrite 선택 시에는 "기존 저장 번호를 백업 파일 내용으로 바꿉니다"로 한국어 라벨을 바꾸고, 현재 데이터 자동 백업을 먼저 제안한다.
- 옵션명도 "테마 설정 적용"보다 "백업 파일의 테마로 바꾸기"처럼 결과 중심으로 바꾸는 편이 낫다.

### 3. 사용자 문구의 기술 용어 집중도

현재 문구 예:

- `시뮬레이션 수`
- `난수 시드`
- `최근 참조 회차`
- `복잡도 지수`
- `프록시`
- `정적 JSON`
- `로컬 최신 회차 업데이트`
- `partial`, `fallback`, `worker`, `dataHealth` 계열 내부 개념

잠재 문제:

- 기능은 강력하지만, 처음 쓰는 사용자는 기본값을 그대로 둬야 하는지 직접 조정해야 하는지 알기 어렵다.
- 고급 설정이 기본 화면에 많이 보이면 "잘못 건드리면 망가질 것 같다"는 인상을 준다.

권장 추가:

- 기본 화면은 "번호 만들기", "번호 추천 받기", "내 번호 확인" 중심으로 단순화한다.
- 고급 설정은 접기/펼치기로 묶고, 기본값 옆에 "그대로 두어도 됩니다"를 짧게 붙인다.
- 개발 용어는 내부 상태/로그에만 남기고, 사용자 화면에는 행동 중심 문장으로 바꾼다.

### 4. `innerHTML` 사용 지점의 장기 관리

현재 구조:

- `Stats`, `Check`, `Ai`, `Backtest`, `DataLists`, `latestDraw`, `dataHealthGate` 등 여러 화면이 template string + `innerHTML`로 렌더링한다.
- 일부 경로는 `escapeHtml()` 또는 `textContent`를 사용한다.
- XSS 회귀 테스트도 이미 일부 존재한다.

잠재 문제:

- 백업 파일, 동기화 payload, 저장된 티켓 id, 날짜, 캠페인 이름처럼 사용자가 가져오거나 외부에서 온 데이터가 화면에 들어가는 경로는 계속 늘어난다.
- 화면마다 escape 방식이 흩어지면 새 기능에서 실수할 가능성이 있다.

권장 추가:

- 외부/저장 데이터가 들어가는 HTML template은 `escapeHtml`, `renderText`, `setText` 같은 공통 헬퍼를 강제한다.
- `innerHTML` 신규 사용을 코드 리뷰 체크리스트와 smoke regression에 넣는다.
- 가능하면 리스트/카드 렌더링은 `document.createElement` 기반 빌더로 점진 전환한다.

### 5. PWA 업데이트를 사용자가 다시 찾기 어려움

현재 구조:

- 새 서비스워커가 준비되면 업데이트 토스트가 뜨고, `업데이트` 버튼으로 `skipWaiting`을 호출한다.
- 토스트 문구와 접근성 속성은 개선되었다.

잠재 문제:

- 사용자가 토스트를 닫으면 "나중에 업데이트"를 어디서 다시 할 수 있는지 찾기 어렵다.
- Android PWA에서는 앱처럼 느껴지기 때문에 사용자는 브라우저 새로고침보다 앱 내부 버튼을 기대한다.

권장 추가:

- 설정 모달에 "앱 업데이트 확인" 또는 "준비된 업데이트 적용" 버튼을 둔다.
- 업데이트가 이미 준비된 상태라면 설정 배지에 "업데이트 가능"을 표시한다.
- 업데이트 적용 후 reload 전에 "저장 중인 내용이 있으면 먼저 저장하세요" 같은 짧은 안내를 검토한다.

## 중간 우선순위 개선 후보

### 6. 모바일 성능과 진행 상태

현재 구조:

- 생성/추천/백테스트는 워커를 사용하고, timeout/fallback 및 취소 흐름이 일부 있다.
- 기본 시뮬레이션 수는 5000이고 최대 20000까지 허용된다.

잠재 문제:

- Android PWA 환경에서는 기기 성능, 배터리 절전, 백그라운드 제한 때문에 워커 응답이 늦어질 수 있다.
- "시뮬레이션 수"를 높이면 더 정확해지는지, 오래 걸리는지 사용자가 바로 이해하기 어렵다.

권장 추가:

- `빠름`, `기본`, `정밀` 같은 프리셋을 먼저 제공하고, 숫자 입력은 고급 설정으로 이동한다.
- 진행 상태에는 "대략 몇 초 더 걸릴 수 있음"보다 "분석 중, 앱을 닫지 마세요"처럼 행동 중심 문구를 쓴다.
- 추천/백테스트 모두 취소 버튼과 완료/중단 상태가 일관되게 보이도록 맞춘다.

### 7. 당첨 보장처럼 보일 위험

현재 구조:

- 추천 화면에는 "당첨을 보장하지 않습니다" 문구가 있다.

잠재 문제:

- 생성 화면과 추천 결과 카드에서는 여전히 "최빈수 우선 추천", "스마트 모드", "AI", "내부 랭킹 점수" 같은 표현이 강한 확신처럼 보일 수 있다.

권장 추가:

- 생성/추천 공통으로 "재미와 참고용입니다. 당첨을 보장하지 않습니다"를 짧게 반복한다.
- "AI 추천"보다 "통계 참고 추천"처럼 기대치를 낮추는 명칭을 검토한다.
- 결과 카드에는 "왜 이 조합이 나왔나요?"를 접힌 설명으로 제공하되, 예측처럼 보이는 표현은 피한다.

### 8. 저장 공간 관리의 다음 단계 부족

현재 구조:

- 저장량 경고, 항목 수 요약, localStorage quota toast가 있다.
- 자동 삭제는 하지 않고 수동 정리를 유도한다.

잠재 문제:

- 사용자가 경고를 봐도 어떤 순서로 정리해야 하는지 모를 수 있다.
- 백업 후 정리, 오래된 히스토리 삭제, 오래된 캠페인 정리 같은 작업이 흩어져 있다.

권장 추가:

- "백업하고 정리하기" 안내 흐름을 하나로 묶는다.
- 정리 전 자동 백업 파일 생성을 제안한다.
- 정리 옵션 예: "오래된 생성 기록만 삭제", "정산 끝난 티켓만 정리", "로컬 최신 회차 보정 데이터 초기화".

### 9. 접근성 알림 범위 확대

현재 구조:

- toast live region이 있고, PWA 업데이트 토스트도 `role="status"`와 `aria-live`를 가진다.
- 백테스트 일부 진행 문구도 `aria-live`가 있다.

잠재 문제:

- 추천 결과, 번호 생성 결과, 당첨 확인 결과, import 결과 같은 큰 화면 변화가 스크린 리더 사용자에게 충분히 전달되지 않을 수 있다.

권장 추가:

- 주요 결과 영역에 `aria-live="polite"` 또는 완료 메시지 전용 live region을 붙인다.
- busy 상태 버튼에는 `aria-busy`나 `disabled` 외에 상태 텍스트를 함께 제공한다.
- "복사", "큐알", "저장" 아이콘 버튼은 결과 맥락을 포함한 aria-label을 검토한다.

### 10. 문서 구조가 개발자 중심으로 길어짐

현재 구조:

- `README.md`는 변경 이력과 기능 설명이 풍부하다.
- `claude.md`는 에이전트 handoff 문서로 유용하다.

잠재 문제:

- 초보 사용자가 README를 열었을 때 "바로 어떻게 쓰는지"보다 변경 이력이 먼저 보인다.
- Windows PowerShell 기본 인코딩으로 UTF-8 문서를 읽으면 한글이 깨져 보일 수 있다.

권장 추가:

- `README.md` 상단을 "3분 사용법"으로 재구성하고, 긴 변경 이력은 아래나 별도 `CHANGELOG.md`로 이동한다.
- `docs/USER_GUIDE.md` 또는 `USER_GUIDE.md`를 추가해 설치, 번호 만들기, 추천 받기, 백업, 데이터 최신화만 쉽게 설명한다.
- `.editorconfig`에 `charset = utf-8`을 추가하면 편집기/문서 인코딩 혼선을 줄일 수 있다.

## 낮은 우선순위 또는 장기 개선

### 11. IndexedDB 전환 검토

현재 저장량은 localStorage로도 관리 가능하지만, 내 번호 보관함/캠페인/히스토리가 커지면 JSON 전체 직렬화 방식은 점점 부담이 된다.

장기적으로는 다음 데이터부터 IndexedDB 전환을 검토할 수 있다.

- 내 번호 보관함
- 캠페인
- 생성 히스토리
- 큰 전략 프리셋 또는 분석 결과 캐시

단, 지금 당장 전환하면 복잡도가 크게 올라가므로 localStorage 안전장치와 백업 흐름을 먼저 다듬는 편이 낫다.

### 12. 사용자 도움말 레이어

현재 앱은 기능 밀도가 높다. 초보자에게는 전체 튜토리얼보다 "각 버튼 옆 한 줄 도움말"이 더 효과적이다.

권장 방향:

- 고급 설정 섹션에 `?` 도움말 버튼을 둔다.
- 처음 설치 후 1회만 "기본값 그대로 번호 만들기" 안내를 보여준다.
- 설정/동기화/백업에는 "잘 모르겠으면 그대로 두세요" 계열의 안심 문구를 제공한다.

## 초보자 친화 문구 개선 방향

### 기본 원칙

- 버튼은 결과를 말한다. 예: "동기화"보다 "최신 회차 확인".
- 라벨은 쉬운 말, 설명은 한 줄, 자세한 기술 설명은 접어 둔다.
- 사용자가 선택해야 하는 경우에는 추천값을 먼저 말한다. 예: "처음이면 기본값을 추천합니다."
- 오류 메시지는 원인보다 다음 행동을 먼저 말한다.
- 고급 사용자를 위한 단어는 괄호나 보조 설명으로 낮춘다.

### 용어 교체 제안

| 현재 표현 | 추천 표현 | 비고 |
| --- | --- | --- |
| 통계 추천 | 번호 추천 또는 통계 참고 추천 | 첫 사용자에게 더 직접적 |
| AI 추천 | 통계 참고 추천 | 과도한 기대 방지 |
| 시뮬레이션 수 | 분석 강도 | 숫자는 고급 설정으로 |
| 난수 시드 | 같은 번호 다시 만들기 코드 | 비워두면 새 번호 강조 |
| 최근 참조 회차 | 최근 몇 회차를 볼까요? | 질문형 라벨이 쉬움 |
| 복잡도 지수 | 번호 섞임 정도 | 고급 설정에 배치 |
| 프록시 | 데이터 연결 주소 | 고급 표시 |
| 정적 JSON | 앱에 포함된 기본 데이터 | 사용자에게 JSON 노출 최소화 |
| 로컬 최신 회차 업데이트 | 내 기기에 임시 저장된 최신 회차 | 의미 중심 |
| 부분 복구 | 일부 데이터만 사용 중 | 사용 가능한 기능도 함께 표시 |
| fallback | 다른 연결 방법으로 재시도 | 로그 외 사용자 화면에서는 영어 제거 |
| Merge | 합치기 | 기존 데이터 유지 |
| Overwrite | 바꾸기 | 기존 데이터 교체 |
| 티켓 목표 회차 | 이 번호를 살 회차 | 로또 맥락에 더 가까움 |
| 티켓북 | 내 번호 보관함 | 사용자가 이해하기 쉬움 |

### 화면별 문구 예시

번호 생성:

추천:

- Android PWA 설치 후 재실행 smoke: 설치 앱 컨텍스트에서 무시드 추천을 여러 번 실행해 번호가 바뀌는지 확인한다.

- dynamic result accessibility test: 생성/추천/확인 결과가 live region 또는 상태 메시지로 전달되는지 확인한다.
